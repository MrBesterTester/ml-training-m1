### Step 2: Create ingest-todo Bridge Skill

Created `.claude/skills/ingest-todo.md` — the core integration piece that bridges the two systems.

This skill:
- Parses a Dylan Davis TODO.md and generates one do-work REQ file per numbered step
- Auto-resolves companion docs (BLUEPRINT, SPECIFICATION) by naming convention
- Embeds the matching Blueprint section in each REQ for full builder context
- Adds custom frontmatter (`source_step`, `source_doc`, `blueprint_ref`, `model_hint`, `batch`)
- Groups REQs by phase, skips already-checked steps, flags manual-review steps
- Creates a UR (User Request) folder with the full TODO as verbatim input

### Step 3: Create start-step and continue-step Skills

Both accept an optional doc-prefix parameter for future document sets (e.g., `/start-step 1.1 v2-upgrade`).

## Architecture Decisions

1. **No nested git repos.** Used `npx skills add` to install do-work from GitHub rather than cloning a repo inside the project.
2. **One REQ per numbered step.** Sub-items within a TODO step are requirements inside the REQ, not separate REQs. The TODO's 14 steps = 14 REQ files.
3. **Blueprint context embedded in REQs.** Each REQ includes the matching Blueprint section so the builder sub-agent has full implementation guidance without needing to read external files.
4. **Manual review preserved.** Steps labeled `[Sam]` still get REQ files but are flagged for human verification.
5. **TODO as source of truth.** The sync-todo skill requires user approval before checking off items — the TODO is never auto-modified.

## Observations

**Sam's note:** The baseline prompts are good general questions about diagnostics and troubleshooting, but the actual test cases for fine-tuning evaluation (not just smoke tests) will need to consist of very specific case data — actual error codes, voltage readings, log output, specific component failures with model numbers, etc.

**Response:** Good point. The baseline responses confirm the model works and give us a "before" snapshot, but the real fine-tuning value will come from specific case data. That will be the focus in Phase 2 when building the training dataset.

The base model also shows clear accuracy gaps that support the case for fine-tuning:
- Boundary scan described as "scanning the board's edges" (incorrect — it's JTAG/IEEE 1149.1)
- Non-ECC memory mislabeled as "Error-Correcting Code"
- Black screen diagnostic suggests measuring PSU voltage on a laptop (unusual)
